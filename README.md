# AtlasAI — Day 1 Delivery

**Goal (per the 7-Day Plan):** empty end-to-end skeleton — Flutter app can hit
the orchestrator through Docker.

This package contains two folders:
- `atlasai_backend/` — FastAPI + Docker + Multi-Agent Orchestrator (5 stub agents)
- `atlasai_app/` — Flutter `lib/` files to drop into your Flutter project

Also included: `SCHEMA.md` — the shared Firestore + Knowledge Graph + API
contract both sides build against for the rest of the week.

---

## 1. Backend — run it

```bash
cd atlasai_backend
cp .env.example .env      # fill in LLM_API_KEY when you have one
docker-compose up --build
```

This starts:
- `atlasai_api` on `http://localhost:8000` (FastAPI)
- `atlasai_chromadb` on `http://localhost:8001` (ChromaDB — not used yet, wired for Day 2)

Verify it's up:
```bash
curl http://localhost:8000/ping
# {"message":"pong from AtlasAI orchestrator"}

curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"query":"What do I check first when Pump 4 vibrates?","user_role":"technician"}'
```
You should get back a stub response from the `maintenance_agent` — the
Orchestrator's keyword routing already works, only the agent reasoning
itself is a stub (built out Day 3/5/6 per the plan).

Swagger docs (auto-generated by FastAPI) are at `http://localhost:8000/docs` —
useful for testing the other agents by forcing them via the `agents` field.

## 2. Flutter — set it up

If you haven't already created the project:
```bash
flutter create atlasai_app
cd atlasai_app
```

Connect Firebase:
```bash
npm install -g firebase-tools   # if not already installed
firebase login
dart pub global activate flutterfire_cli
flutterfire configure --project=atlasai-hackathon
```
This generates the **real** `lib/firebase_options.dart` — it will overwrite
the placeholder included in this package (that's expected).

Add dependencies:
```bash
flutter pub add firebase_core firebase_auth cloud_firestore firebase_storage firebase_messaging http
```

Copy this package's `atlasai_app/lib/` contents into your project's `lib/`
folder (merge, don't overwrite `firebase_options.dart` after you've run
`flutterfire configure`).

Run it:
```bash
flutter run
```

### Point Flutter at your backend

`lib/services/orchestrator_service.dart` defaults to `http://10.0.2.2:8000`,
which is correct **only for the Android emulator** (it maps to your host
machine's `localhost`). Adjust based on what you're running on:

| Target | baseUrl |
|---|---|
| Android emulator | `http://10.0.2.2:8000` (default, no change needed) |
| iOS simulator | `http://localhost:8000` |
| Physical device (same WiFi) | `http://<your-machine-LAN-IP>:8000` |
| Web (Chrome) | `http://localhost:8000` |

### What you should see

`main.dart` launches a **Day 1 Status Screen** that:
1. Confirms Firebase initialized ("Firebase connected ✅")
2. Pings the FastAPI orchestrator through Docker ("Backend reachable ✅")
3. Has a button through to the role-based `HomeShell` stub (technician view by default)

If both checks are green, Day 1 is done — the skeleton is proven end-to-end.

## 3. Enable Firebase services (console)

In [Firebase Console](https://console.firebase.google.com) → your project:
- **Authentication** → Sign-in method → enable Email/Password
- **Firestore Database** → Create database → test mode (tighten rules before submission)
- **Storage** → Get started → test mode
- **Cloud Messaging** → no manual setup needed
